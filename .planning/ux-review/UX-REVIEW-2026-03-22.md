# Airport Digital Twin — UX Review Report

**Date:** 2026-03-22
**Reviewer:** Claude (acting as UX designer)
**App URL:** https://airport-digital-twin-dev-7474645572615955.aws.databricksapps.com
**Airports Tested:** KSFO (San Francisco), EGLL (London Heathrow), RJTT (Tokyo Haneda)

---

## Executive Summary

The Airport Digital Twin delivers a compelling, data-rich experience across 2D and 3D views with realistic airline distributions, weather integration, and multi-airport support. The core simulation is solid — flights move through proper lifecycle phases (parking, pushback, taxi, takeoff, enroute, approach, landing), gate turnarounds work, and airline/route assignments are airport-appropriate. However, several UX refinements would elevate the experience from "impressive demo" to "production-grade digital twin."

**Overall Grade: B+** — Strong foundation, needs polish on visual fidelity and edge cases.

---

## 1. 2D View Assessment

### What Works Well
- **Satellite imagery** provides excellent spatial context — runways, taxiways, terminal buildings all clearly visible
- **Flight state color coding** is intuitive (green=parked, orange=taxi, red=departing, cyan=approaching, yellow=enroute)
- **Flight list sidebar** is well-organized with callsign, altitude, speed, and state indicators
- **Flight detail panel** is comprehensive: gate assignment, origin/destination, delay prediction, gate recommendations, baggage tracking, turnaround phases
- **Trajectory polylines** show smooth paths with 80+ waypoints per flight
- **Gate labels** visible at appropriate zoom levels
- **Phase filter bar** at the top allows filtering by flight state — useful for focused analysis

### Issues Found

| # | Issue | Severity | Description |
|---|-------|----------|-------------|
| 1 | **Aircraft silhouette size** | Medium | At intermediate zoom, aircraft silhouettes appear oversized relative to terminal buildings. A parked B738 visually spans nearly the width of the terminal building. Real aircraft wingspan (35m for B738) should be ~1/3 the width of a terminal concourse. |
| 2 | **Gate nose orientation** | Medium | Most parked aircraft face roughly toward terminal walls, but alignment is approximate — several aircraft appear rotated 10-20° off the perpendicular. Wings are not always parallel to the building face. |
| 3 | **NaN altitude/speed flights** | High | Two flights (hex codes B758F7, D222AE) show `ALT: NaNft SPD: NaNkts` in the sidebar. These appear to be raw ICAO hex addresses used as callsigns, with missing numeric data — likely a synthetic data generation edge case. |
| 4 | **Duplicate callsigns** | Medium | At RJTT, `JAL765` appears twice — once as PKD (parked) and once as ENR (enroute). The simulation should enforce unique active callsigns. |
| 5 | **Taxi path realism** | Low | Taxi trajectories show reasonable paths but don't strictly follow OSM taxiway centerlines. Some paths cut through grass/apron areas rather than following marked taxiways. |
| 6 | **Map click on aircraft** | Low | Clicking directly on aircraft markers on the map sometimes fails to select (timeout). The sidebar flight list buttons work reliably as an alternative. |

## 2. 3D View Assessment

### What Works Well
- **Terminal buildings** rendered as extruded 3D blocks with appropriate heights — gives immediate sense of airport scale
- **Aircraft 3D models** (GLB) load correctly — the A320 model is properly proportioned after the recent fix
- **Camera controls** (orbit, zoom, pan) are smooth and responsive
- **Top-down view** aligns well with 2D satellite imagery, confirming spatial accuracy
- **Trajectory lines** visible in 3D, showing altitude changes during approach/departure
- **Reset view button** works reliably to return to default camera position

### Issues Found

| # | Issue | Severity | Description |
|---|-------|----------|-------------|
| 7 | **Aircraft scale in 3D** | Medium | Similar to 2D, aircraft models appear slightly oversized when viewed close to terminal buildings. The ratio of aircraft to building size doesn't feel quite right at close zoom. |
| 8 | **Terminal building detail** | Low | Buildings are uniform extruded blocks — no roof variation, no jetway/finger structures, no windows or texture. Adequate for overview but lacks detail at close range. |
| 9 | **WebGL context lost** | Low | Console shows `THREE.WebGLRenderer: Context Lost` when switching between 2D and 3D views. The 3D view recovers correctly but this indicates resource management could be tighter. |
| 10 | **2D↔3D state not shared** | Medium | When selecting a flight in 2D and switching to 3D, the selection is lost. Camera position/zoom area is also not synchronized between views. |

## 3. Airport Operations Assessment

### What Works Well
- **Gate allocation** works with proper terminal/gate numbering per airport (A-G at KSFO, numbered terminals at EGLL, T1-T9 at RJTT)
- **Gate status panel** shows available vs. occupied counts per terminal — useful operational view
- **Turnaround phases** are modeled: docking → unloading → cleaning → loading → boarding → pushback
- **Gate recommendations** show scored alternatives when aircraft is on approach (proximity, size compatibility, availability)
- **Delay predictions** with confidence percentages (On Time 89%, Moderate Delay, etc.)
- **Baggage tracking** with connecting bag risk alerts
- **FIDS panel** shows realistic arrival/departure boards

### Issues Found

| # | Issue | Severity | Description |
|---|-------|----------|-------------|
| 11 | **Gate turnover visibility** | Medium | Hard to observe aircraft leaving gates for new destinations in real-time. The simulation updates happen, but there's no visual notification or animation of pushback. Would benefit from a "recently departed" indicator or pushback animation. |
| 12 | **FIDS limited info** | Low | FIDS board shows flight basics but doesn't show delay status or gate changes — standard FIDS features passengers expect. |

## 4. Multi-Airport & Switching Assessment

### What Works Well
- **29 airports available** in dropdown — extensive coverage
- **Airline distributions are airport-appropriate:**
  - KSFO: UAL, VRD, SWA, AAL dominant (correct for SFO hub)
  - EGLL: BAW, VIR, EZY dominant (correct for Heathrow)
  - RJTT: ANA, JAL dominant (correct for Haneda)
- **Weather updates per airport:** 19°C at SFO, 11°C at LHR, 20°C at HND — pulled from METAR
- **Terminal structures adapt per airport** — not hardcoded
- **Gate counts scale appropriately:** 87 gates at KSFO, ~80 at EGLL, 87 at RJTT

### Switching Performance

| Route | Measured Time | User Experience |
|-------|--------------|-----------------|
| KSFO → EGLL | ~18.7 seconds | Acceptable for demo, but feels slow. Progress overlay helps. |
| EGLL → RJTT | ~15-20 seconds | Similar range. Satellite tile loading adds perceived delay. |

### Issues Found

| # | Issue | Severity | Description |
|---|-------|----------|-------------|
| 13 | **Switch time too long** | High | 15-20 seconds per airport switch is a significant UX friction. Users expect near-instant transitions. The progress overlay mitigates but doesn't solve the core issue. Pre-caching adjacent airports or lazy-loading would help. |
| 14 | **No switch cancel** | Low | Once an airport switch starts, there's no way to cancel it. If a user accidentally clicks the wrong airport, they must wait for the full load. |

## 6. Prioritized Improvement Plan

### Phase 1: Critical Fixes (High Impact, Low Effort)

1. **Fix NaN flights bug** — Validate all synthetic data fields (altitude, speed, callsign) before sending to frontend. Reject or filter flights with missing numeric data. Filter out raw hex ICAO addresses as callsigns.

2. **Enforce unique callsigns** — Add a constraint in the synthetic data generator to prevent duplicate active callsigns (e.g., the JAL765 duplicate at RJTT).

3. **Form accessibility** — Add `id` or `name` attributes to the 2 form fields flagged in console.

### Phase 2: Visual Fidelity (Medium Impact, Medium Effort)

4. **Scale aircraft silhouettes** — Reduce 2D SVG and 3D model scale by ~40% to match realistic aircraft-to-building proportions. A B738 wingspan (35m) should appear appropriately small next to a terminal concourse (100-200m wide).

5. **Improve gate alignment** — Snap parked aircraft orientation to perpendicular to the nearest terminal wall face. Ensure wings are parallel to building edges. This may require computing the wall normal vector from OSM building geometry.

6. **2D↔3D state sync** — Share selected flight and camera viewport between 2D and 3D views. When switching views, maintain the user's focus area and selection.

### Phase 3: Performance (High Impact, High Effort)

7. **Reduce airport switch time** — Target: <5 seconds. Approaches:
   - Pre-cache the 5-10 most popular airports in background
   - Lazy-load satellite tiles (show vector map first, then overlay imagery)
   - Parallelize OSM fetch, flight generation, and ML model loading
   - Use the existing pre-load plan (`.planning/phases/13-airport-preload-cache/PLAN.md`)

8. **WebGL resource management** — Properly dispose Three.js scene, geometries, materials, and textures when switching to 2D view to prevent context loss events.

### Phase 4: Operational Realism (Medium Impact, High Effort)

9. **Taxiway-following paths** — Route taxi trajectories along OSM taxiway centerlines using graph-based pathfinding (A* or Dijkstra on taxiway network). This replaces the current straight-line interpolation with realistic ground movement.

10. **Pushback animation** — Animate aircraft pushing back from gate before taxi-out. Add a brief pushback phase where the aircraft moves backward away from the terminal, then turns toward the taxiway. This makes gate turnovers visible and satisfying.

11. **FIDS enhancements** — Add delay status indicators, gate change notifications, and estimated times to the FIDS board. Include both arrivals and departures with filters.

12. **Map marker click reliability** — Investigate why direct clicks on aircraft map markers sometimes timeout. May need larger hit areas or event delegation.

### Phase 5: Polish (Low Impact, Low Effort)

13. **Airport switch cancel button** — Add a "Cancel" button to the progress overlay that aborts the switch and returns to the previous airport.

14. **Terminal building detail** — Add variation to 3D building extrusions (roof shapes, different heights for concourses vs. main terminal). Add jetway/finger geometries extending from gates.

15. **Recently departed indicator** — Show a brief "just departed" visual cue when an aircraft pushes back from a gate, making turnarounds more visible in the 2D view.
