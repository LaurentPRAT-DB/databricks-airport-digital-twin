# UX Audit — Airport Digital Twin

**Date:** 2026-03-20
**Version:** v0.1.0, Build #268
**Auditor:** Claude (acting as UX designer)
**Airports tested:** EGLL (London Heathrow), LSGG (Geneva), KJFK (New York JFK)

---

## Executive Summary

The application provides a compelling airport digital twin experience with real-time flight tracking, 3D visualization, gate management, and ML predictions. The 2D/3D views are functional and visually coherent. However, several realism and performance issues reduce credibility for domain-knowledgeable users.

**Overall Rating:** 7/10 — Functional and impressive for a demo, but needs data realism fixes to be credible to aviation professionals.

---

## Findings

### CRITICAL — Realism Bugs

#### Finding 1: US-centric airlines at non-US airports
**Severity:** CRITICAL
**Where:** LSGG (Geneva)
**Issue:** ~80% of flights are US carriers (AAL, SWA, JBU, ASA, DAL, UAL). Southwest (SWA) and JetBlue (JBU) have never operated at Geneva. Expected: Swiss (SWR), EasyJet (EZY), Lufthansa (DLH), Air France (AFR), British Airways (BAW).
**Root cause:** The airline pool in the fallback/schedule generator draws from a hardcoded US-centric list when no calibration profile exists. The calibration system works well for profiled airports (KJFK has correct JBU/DAL/AAL/IBE/KLM mix) but LSGG lacks a profile.
**Fix:** Add per-region airline pools (European, Asian, Middle Eastern, African) and select airlines based on the airport's country/region. Alternatively, expand calibration profiles to cover all cached airports.

#### Finding 2: Unrealistic approach speeds
**Severity:** CRITICAL
**Where:** EGLL — AFR2663 descending
**Issue:** 372 kts at 1206 ft altitude. Real approach speed at that altitude should be 140-160 kts (Vref). Even fast approaches rarely exceed 180 kts below 3000 ft.
**Root cause:** Speed deceleration curve in the synthetic trajectory generator doesn't constrain speed based on altitude during approach phase.
**Fix:** Enforce speed envelope: below 3000 ft AGL, cap at 180 kts; below 1500 ft, cap at 160 kts; below 500 ft, cap at Vref (~140 kts for A330).

#### Finding 3: US routes at Geneva
**Severity:** HIGH
**Where:** LSGG — AAL100
**Issue:** Origin: JFK, Destination: DEN. While JFK→GVA is a plausible transatlantic route, GVA→DEN is not. American Airlines doesn't operate at Geneva.
**Root cause:** The proximity-based airport selection fix (deployed this session) correctly picks nearby airports for destinations, but the *airline* assigned doesn't match the route. The fallback airline selection is decoupled from route logic.
**Fix:** Tie airline selection to route plausibility. If origin/destination pair has a realistic carrier, use it.

### HIGH — Performance

#### Finding 4: Airport switch takes ~16s
**Severity:** HIGH
**Where:** All airport switches
**Measurements:**
- EGLL → LSGG: **16.8s**
- LSGG → KJFK: **16.3s**

**Root cause:** Known issue — Lakebase/UC cache broken in app context, so every switch falls through to OSM Overpass API fetch (10-16s).
**Impact:** Users lose engagement during 16s wait. The loading spinner helps but the wait is excessive.
**Fix:** Restore Lakebase/UC caching (blocked by OAuth issue) or implement client-side OSM response caching in IndexedDB.

### MEDIUM — Data Quality

#### Finding 5: Some flights have no gate assignment
**Severity:** MEDIUM
**Where:** EGLL — multiple flights show `assigned_gate: null`
**Issue:** Ground flights without gate assignments appear to float on the map without a clear position anchor.
**Fix:** Ensure all ground-phase flights get a gate assignment or at minimum a stand/apron position.

#### Finding 6: Weather data is static
**Severity:** LOW
**Where:** All airports
**Issue:** Weather shows "15°C 279@10kt 10SM" regardless of airport or time. Geneva and JFK both show identical weather.
**Fix:** Fetch real METAR data per airport ICAO code, or at minimum vary synthetic weather by region and season.

### POSITIVE — What Works Well

#### Finding 7: Calibrated airports have excellent airline realism
**Where:** KJFK
**Detail:** JFK shows JBU (JetBlue, T5 hub), DAL (Delta, T2/T4), AAL, IBE, KLM, DLH, TAM, AMX, ENY, EDV, PDT, SKW, RPA, FFT — all realistic JFK operators including regional affiliates. The calibration system delivers impressive fidelity.

#### Finding 8: Gate management is functional and informative
**Where:** All airports
**Detail:** Terminal tabs, occupancy counts (e.g., "Terminal B: 22 free | 8 used / 30"), turnaround progress with 7 steps, active equipment list (belt loader, ground power), estimated departure time, baggage status with connecting bag risk alerts — all working correctly.

#### Finding 9: 2D/3D coherence is good
**Where:** KJFK — AAL100 selected in both views
**Detail:** Same flight shows consistent position at gate B44 in both 2D and 3D. Trajectory lines visible in both views. 3D terminal buildings, runways, and taxiways match 2D OSM overlay. No position drift or misalignment observed.

#### Finding 10: Flight detail panel is comprehensive
**Where:** All airports
**Detail:** Callsign, ICAO hex, phase, origin/destination, aircraft type, position (lat/lon/alt), movement (speed/heading/vrate), delay prediction with confidence %, gate recommendations with scores and taxi times, baggage status with connecting flight risk alerts. Well-organized with clear sections.

#### Finding 11: Console is clean
**Where:** All airports, both views
**Detail:** Zero console errors or warnings. All API calls return 200. WebSocket connection stable ("Connected" indicator green).

#### Finding 12: OSM gate loading works across airports
**Where:** EGLL (142 gates), LSGG (60 gates), KJFK (137 gates)
**Detail:** Each airport loads distinct gate configurations from OSM with correct terminal groupings. EGLL: T1-T5 + A/B/C. LSGG: PP/A/B/C/D/F. KJFK: T1-T9 + A/B. Gate occupancy tracking adjusts dynamically.

## Improvement Plan (Priority Order)

### Phase 1: Realism Fixes (High Impact, Moderate Effort)

1. **Per-region airline pools** — Create airline dicts by region (Europe, Asia, Middle East, Americas). When no calibration profile exists, select airlines from the airport's region. Estimated: 2-3 hours.

2. **Speed envelope on approach** — Add altitude-based speed constraints to `generate_synthetic_trajectory()`. Below 3000ft: max 180kts. Below 1500ft: max 160kts. Below 500ft: max Vref (type-dependent). Estimated: 1-2 hours.

3. **Expand calibration profiles** — Add profiles for all 29 cached airports (currently ~33 US + major international airports have profiles). Priority: LSGG, LGAV, OMAA, OMDB, FAOR, GMMN. Estimated: 3-4 hours.

### Phase 2: Performance (High Impact, Blocked)

4. **Fix Lakebase/UC caching** — Resolve OAuth issue in Databricks App context so airport configs load from UC/Lakebase instead of OSM every time. Would reduce switch time from ~16s to <3s. Currently blocked by platform limitation.

5. **Client-side OSM cache** — As a workaround, cache OSM responses in browser IndexedDB/localStorage. On repeat visits, load from cache. Estimated: 2-3 hours.

### Phase 3: Polish (Medium Impact, Low Effort)

6. **Per-airport weather** — Fetch METAR from aviationweather.gov or generate region-appropriate synthetic weather. Estimated: 1-2 hours.

7. **Gate assignment guarantee** — Ensure all ground-phase flights have a gate or stand position. Estimated: 1 hour.

8. **Route-airline coupling** — When picking airlines for a route, cross-reference that the airline actually operates between the origin and destination regions. Estimated: 2-3 hours.
