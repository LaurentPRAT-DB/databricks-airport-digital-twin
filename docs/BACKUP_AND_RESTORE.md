# Airport Digital Twin — Backup & Restore Guide

This document explains how to create a portable backup of the Airport Digital Twin application and deploy it to a different Databricks workspace or restore it offline.

---

## Architecture Overview

The application spans several Databricks services. Understanding what lives where is essential for a complete backup.

```
+---------------------------+     +---------------------------+
|     Git Repository        |     |   Databricks Workspace    |
|                           |     |                           |
|  app/backend/   (FastAPI) |     |  UC Catalog.Schema        |
|  app/frontend/  (React)   |     |    flight_status_gold     |
|  src/           (Core)    |     |    flight_positions_history|
|  data/profiles/ (JSON)    |     |    airport_profiles        |
|  resources/     (DABs)    |     |                           |
|  databricks/    (DLT)     |     |  UC Volume: static_assets |
|                           |     |    models/aircraft/*.glb   |
|  app.yaml  (env config)   |     |                           |
|  databricks.yml (bundle)  |     |  Lakebase (PostgreSQL)    |
|                           |     |    flight_status           |
+---------------------------+     |    weather_observations    |
                                  |    baggage_status          |
                                  |                           |
                                  |  SQL Warehouse             |
                                  |  DLT Pipeline              |
                                  |  Genie Space (optional)    |
                                  |  Databricks App            |
                                  +---------------------------+
```

### What is workspace-bound vs portable

| Component | Portable? | Notes |
|-----------|:---------:|-------|
| Source code (git repo) | Yes | Self-contained |
| Calibration profiles (`data/calibration/profiles/*.json`) | Yes | 33 airport profiles, checked into repo |
| Frontend build (`app/frontend/dist/`) | Yes | Rebuilt from source |
| 3D models (`.glb` files) | **No** | Stored in UC Volumes (25MB+ files exceed bundle limits) |
| UC Delta tables | **No** | Regenerated by DLT pipeline or sync scripts |
| Lakebase PostgreSQL | **No** | Recreated via schema DDL + sync jobs |
| SQL Warehouse ID | **No** | Workspace-specific resource |
| Lakebase endpoint | **No** | Workspace-specific hostname |
| Genie Space ID | **No** | Workspace-specific resource |
| App OAuth credentials | **No** | Auto-generated per workspace |

## Creating a Backup

### Prerequisites

- Databricks CLI installed and configured
- Access to the current workspace (profile `FEVM_SERVERLESS_STABLE` by default)
- ~200MB free disk space for the tarball

### Quick Backup

```bash
cd /path/to/databricks_airport_digital_twin
./scripts/backup.sh
```

This creates `airport-digital-twin-backup-YYYYMMDD-HHMMSS.tar.gz` in the repo root.

### Custom Backup

```bash
./scripts/backup.sh \
  --profile FEVM_SERVERLESS_STABLE \
  --catalog serverless_stable_3n0ihb_catalog \
  --schema airport_digital_twin \
  --output ~/backups/adt-backup.tar.gz
```

### What the Backup Contains

```
airport-digital-twin-backup-*/
  MANIFEST.md                         # This file describes the backup contents
  repo/                               # Full source code snapshot
    app/backend/                      # FastAPI backend
    app/frontend/dist/                # Built frontend (without 3D models)
    src/                              # Core simulation, ML, calibration
    data/calibration/profiles/        # 33 airport calibration profiles
    scripts/                          # Includes backup.sh + bootstrap.sh
    resources/                        # DABs job/pipeline YAML configs
    databricks/                       # DLT pipeline notebooks
    app.yaml                          # Current app config
    databricks.yml                    # Current bundle config
    ...
  uc_volumes/
    models/aircraft/                  # All 3D model files (.glb)
      boeing-737.glb                  #   34 KB
      airbus_a320.glb                 #   424 KB
      airbus_a380.glb                 #   393 KB
      cathay_pacific_airbus_a330-300.glb  # 1.1 MB
      pan_am_airbus_a310-324.glb      #   3.0 MB
      fighter_jet_collection.glb      #   25 MB
      ...                             #   11 files total
  lakebase/
    lakebase_schema.sql               # PostgreSQL DDL for all tables
    setup_lakebase.py                 # Automated Lakebase setup script
  uc_tables/
    flight_status_gold.sql            # Delta table DDL (informational)
    flight_positions_history.sql      # Delta table DDL (informational)
    airport_profiles.sql              # Delta table DDL (informational)
    dlt_pipeline_config.json          # DLT pipeline configuration
  config/
    app.yaml.template                 # app.yaml with ${PLACEHOLDER} variables
    databricks.yml.template           # databricks.yml with ${PLACEHOLDER} variables
```

### Offline Storage

The tarball is fully self-contained. Store it on:
- External drive / USB stick
- Cloud storage (S3, ADLS, GCS)
- Internal artifact repository

No Databricks connectivity is needed to store or transfer the backup. Internet access is only required during restore (to reach the target workspace).

## Restoring to a Different Workspace

### Prerequisites on the Target Workspace

Before running bootstrap, ensure:

1. **Databricks CLI** configured with a profile for the target workspace
2. **SQL Warehouse** created and running (you need its ID)
3. **Unity Catalog** enabled on the workspace
4. **Serverless compute** available (for DLT pipelines and test jobs)
5. **Node.js 18+** installed locally (for frontend build)
6. **Python 3.10+** with `uv` (for backend dependencies)

### Quick Restore

```bash
# Extract the backup
tar xzf airport-digital-twin-backup-*.tar.gz
cd airport-digital-twin-backup-*/repo

# Deploy to target workspace
./scripts/bootstrap.sh \
  --profile TARGET_WORKSPACE \
  --warehouse-id <your-warehouse-id>
```

The script auto-detects the catalog and configures everything.

### Full Restore with All Options

```bash
./scripts/bootstrap.sh \
  --profile TARGET_WORKSPACE \
  --catalog my_catalog \
  --schema airport_digital_twin \
  --warehouse-id abc123def456 \
  --lakebase-project airport-digital-twin \
  --target prod \
  --backup-dir ..
```

### What Bootstrap Does (Step by Step)

```
Step 1/7  Create UC schema
          └── Creates {catalog}.airport_digital_twin if it doesn't exist

Step 2/7  Upload 3D models to UC Volume
          ├── Creates Volume: {catalog}.airport_digital_twin.static_assets
          └── Uploads 11 GLB files to /models/aircraft/

Step 3/7  Set up Lakebase (skippable with --skip-lakebase)
          ├── Creates Lakebase project: airport-digital-twin
          └── Records endpoint hostname for app.yaml

Step 4/7  Genie Space (manual step, skippable with --skip-genie)
          └── Prints instructions for manual creation via UI

Step 5/7  Patch configuration files
          ├── app.yaml:       replaces host, warehouse ID, catalog, Lakebase, Genie
          ├── databricks.yml: adds target with profile + catalog
          └── resources/app.yml: replaces warehouse ID

Step 6/7  Build frontend
          └── npm install && npm run build

Step 7/7  Deploy and start
          ├── databricks bundle deploy --target {target}
          └── databricks apps start airport-digital-twin-{target}
```

### Dry Run

Preview what will happen without making changes:

```bash
./scripts/bootstrap.sh \
  --profile TARGET_WORKSPACE \
  --warehouse-id abc123 \
  --dry-run
```

## Post-Restore Steps

### 1. Verify the App is Running

Wait ~2 minutes after bootstrap completes, then visit the app URL printed in the output.

The app starts in **demo mode** by default (`DEMO_MODE=true`), which generates synthetic flight data without needing Delta tables or Lakebase. This means the app is functional immediately after deploy.

### 2. Set Up Data Pipeline (Optional)

To enable real data flow (Delta tables + Lakebase):

```bash
# Run the DLT pipeline to create Bronze/Silver/Gold tables
databricks bundle run airport_dlt_pipeline --target dev

# Sync calibration profiles to UC
databricks bundle run profile_sync --target dev

# Set up Lakebase tables and start syncing
python scripts/setup_lakebase.py --profile TARGET_WORKSPACE
python scripts/sync_delta_to_lakebase.py --profile TARGET_WORKSPACE
```

### 3. Create a Genie Space (Optional)

The Airport Operations Assistant chatbot requires a Genie Space:

1. Go to the target workspace UI
2. Navigate to **AI/BI > Genie Spaces**
3. Create a new space connected to `{catalog}.airport_digital_twin`
4. Copy the Space ID from the URL
5. Update `app.yaml`:
   ```yaml
   - name: GENIE_SPACE_ID
     value: "<your-new-space-id>"
   ```
6. Redeploy: `databricks bundle deploy --target dev`

### 4. Run Tests

```bash
# Unit tests on serverless
databricks bundle run unit_test --target dev

# E2E smoke tests against live endpoints
databricks bundle run e2e_smoke_test --target dev
```

## Deploying from Git (Without Backup)

If you have the git repo but no backup tarball, you can still deploy to a new workspace. The only missing piece is the 3D model files.

```bash
# Clone the repo
git clone https://github.com/LaurentPRAT-DB/databricks-airport-digital-twin.git
cd databricks-airport-digital-twin

# Option A: Copy GLB files from another machine
scp user@source:~/models/*.glb app/frontend/public/models/aircraft/

# Option B: Deploy without 3D models (2D map still works, 3D uses fallback geometry)
./scripts/bootstrap.sh --profile TARGET --warehouse-id abc123
```

| `app.yaml` | `DATABRICKS_CATALOG` | `my_catalog` |

## Troubleshooting

### Deployment fails with "File size exceeded max size"

A 3D model file is being included in the bundle. Ensure `app/frontend/dist/models/**` is in the `exclude` list in `databricks.yml`. Models are served from UC Volumes at runtime.

### 3D models not loading after restore

Check that the UC Volume was created and populated:
```bash
databricks fs ls "dbfs:/Volumes/<catalog>/<schema>/static_assets/models/aircraft" --profile TARGET
```

If empty, re-upload from the backup:
```bash
for f in ../uc_volumes/models/aircraft/*.glb; do
  databricks fs cp "$f" "dbfs:/Volumes/<catalog>/<schema>/static_assets/models/aircraft/$(basename $f)" \
    --profile TARGET --overwrite
done
```

### App starts but shows no flights

The app is likely in demo mode and waiting for the simulation to initialize. Check logs:
```bash
databricks apps get-logs airport-digital-twin-dev --profile TARGET 2>&1 | tail -50
```

### Lakebase connection fails

Lakebase endpoints are workspace-specific. Verify the endpoint exists:
```bash
databricks postgres get-endpoint \
  "projects/airport-digital-twin/branches/production/endpoints/primary" \
  --profile TARGET
```

If it doesn't exist, either create it (`scripts/setup_lakebase.py`) or set `DEMO_MODE=true` in `app.yaml` to bypass Lakebase entirely.
